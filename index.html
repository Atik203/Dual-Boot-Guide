3
Restructure EFI partition via Diskpart to bypass 100MB limit
INSTALL SETUP ▶

By default, Windows allocates a tiny 100MB EFI system partition (ESP). This fills up rapidly during Linux kernel upgrades. Because the 16MB Microsoft Reserved (MSR) partition directly blocks the ESP, it cannot be expanded later. A clean restructure is required during Windows installation.

  1. Boot the Windows 11 installation media USB.
  2. On the welcome setup screen (before clicking Install), press Shift + F10 to summon the command console.
  3. Enter the low-level diskpart interface:
WinPE Command Prompt
diskpart
# List physical drives to find primary NVMe
list disk
# Select NVMe drive (usually disk 0)
select disk 0
# Clear existing OEM partition table (WARNING: erases drive!)
clean
# Create oversized EFI System Partition (ESP)
create partition efi size=500
format quick fs=fat32 label="System"
# Create Microsoft Reserved partition
create partition msr size=16
exit

After exiting CMD, continue the graphical setup. Select the "Unallocated Space" to install Windows. It will automatically leverage the 500MB EFI partition you just provisioned.

6
Decrypt or Suspend Windows BitLocker Drive Encryption
SAFETY ▶

BitLocker relies on the TPM (Trusted Platform Module) to release the decryption key by measuring the system boot states (PCR registers). Modifying partition layouts or introducing GRUB boot sequences invalidates these PCR hashes, permanently locking you out of Windows with a BitLocker Recovery screen.

PowerShell (Admin)
# Check BitLocker encryption status on all drives
manage-bde -status

# Fully disable BitLocker decryption on system drive (highly recommended)
manage-bde -off C:
⚠️
If BitLocker is ON: Wait for full decryption (check status with `manage-bde -status C:` until percentage reaches 0%). Moving/resizing an active BitLocker partition will destroy the header. Retrieve your 48-digit Recovery Key online via https://account.microsoft.com/devices/recoverykey beforehand.

Back up SSH keys and Git credentials FIRST
CRITICAL ▶

SSH keys are the most commonly forgotten thing. Back these up or you'll lose GitHub access.

PowerShell
# See your SSH keys
dir "C:\Users\YourName\.ssh"

# Copy the entire .ssh folder to E: first (as staging area)
Copy-Item -Recurse "C:\Users\YourName\.ssh" "E:\Backup_SSH\"

# Also export Git global config
Copy-Item "C:\Users\YourName\.gitconfig" "E:\Backup_SSH\gitconfig_backup"
⚠️
Files to look for in .ssh: id_rsa, id_ed25519, id_ed25519.pub, config, known_hosts. The .pub files are public (safe). The others without .pub are private — treat them like passwords.
4
Copy E: drive to phone (65GB transfer)
~1–3 hours ▶

Use robocopy for reliable large file transfers — it resumes if interrupted, unlike drag-and-drop which restarts from zero on failure.

PowerShell (Admin)
# Replace X with your phone's drive letter in File Explorer
# (e.g., if phone shows as F:, use F:)
# First, find phone drive letter:
Get-PSDrive | Where-Object Provider -like "*FileSystem*"

# Robust copy command — copies E: to phone
robocopy E:\ "X:\LaptopBackup_2026\E_Drive_Dev\" /E /COPYALL /R:3 /W:5 /LOG:"C:\robocopy_log.txt" /TEE

# Flags explained:
# /E = copy subdirectories including empty ones
# /COPYALL = copy all file attributes
# /R:3 = retry 3 times on failure
# /W:5 = wait 5 seconds between retries
# /LOG = write log file for verification
# /TEE = show progress in console AND log
💡
MTP (phone USB) transfer speed is typically 15–40 MB/s. At 65GB, expect 30–70 minutes. Keep the phone screen on (set Display Timeout to "Never" in phone settings during transfer).
⚠️
Do NOT lock the phone or let it sleep during transfer — MTP connection drops on sleep. On Vivo: Settings → Display → Screen Timeout → set to "30 minutes" or max.

6
Verify backup integrity
VERIFY ▶
PowerShell
# Count files in E: drive
(Get-ChildItem "E:\" -Recurse -File -ErrorAction SilentlyContinue).Count

# Count files in phone backup (replace X:)
(Get-ChildItem "X:\LaptopBackup_2026\E_Drive_Dev\" -Recurse -File -ErrorAction SilentlyContinue).Count

# Check total size of backup
(Get-ChildItem "X:\LaptopBackup_2026\" -Recurse | Measure-Object Length -Sum).Sum / 1GB

File counts should match (or be very close). If there's a big discrepancy, check the robocopy log at C:\robocopy_log.txt for errors.
