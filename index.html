3
Restructure EFI partition via Diskpart to bypass 100MB limit
INSTALL SETUP ▶

By default, Windows allocates a tiny 100MB EFI system partition (ESP). This fills up rapidly during Linux kernel upgrades. Because the 16MB Microsoft Reserved (MSR) partition directly blocks the ESP, it cannot be expanded later. A clean restructure is required during Windows installation.

  1. Boot the Windows 11 installation media USB.
  2. On the welcome setup screen (before clicking Install), press Shift + F10 to summon the command console.
  3. Enter the low-level diskpart interface:
WinPE Command Prompt
diskpart
# List physical drives to find primary NVMe
list disk
# Select NVMe drive (usually disk 0)
select disk 0
# Clear existing OEM partition table (WARNING: erases drive!)
clean
# Create oversized EFI System Partition (ESP)
create partition efi size=500
format quick fs=fat32 label="System"
# Create Microsoft Reserved partition
create partition msr size=16
exit

After exiting CMD, continue the graphical setup. Select the "Unallocated Space" to install Windows. It will automatically leverage the 500MB EFI partition you just provisioned.

6
Decrypt or Suspend Windows BitLocker Drive Encryption
SAFETY ▶

BitLocker relies on the TPM (Trusted Platform Module) to release the decryption key by measuring the system boot states (PCR registers). Modifying partition layouts or introducing GRUB boot sequences invalidates these PCR hashes, permanently locking you out of Windows with a BitLocker Recovery screen.

PowerShell (Admin)
# Check BitLocker encryption status on all drives
manage-bde -status
 
# Fully disable BitLocker decryption on system drive (highly recommended)
manage-bde -off C:
⚠️
If BitLocker is ON: Wait for full decryption (check status with `manage-bde -status C:` until percentage reaches 0%). Moving/resizing an active BitLocker partition will destroy the header. Retrieve your 48-digit Recovery Key online via https://account.microsoft.com/devices/recoverykey beforehand.

4
Copy E: drive to phone (65GB transfer)
~1–3 hours ▶

Use robocopy for reliable large file transfers — it resumes if interrupted, unlike drag-and-drop which restarts from zero on failure.

PowerShell (Admin)
# Replace X with your phone's drive letter in File Explorer
# (e.g., if phone shows as F:, use F:)
# First, find phone drive letter:
Get-PSDrive | Where-Object Provider -like "*FileSystem*"

# Robust copy command — copies E: to phone
robocopy E:\ "X:\LaptopBackup_2026\E_Drive_Dev\" /E /COPYALL /R:3 /W:5 /LOG:"C:\robocopy_log.txt" /TEE

# Flags explained:
# /E = copy subdirectories including empty ones
# /COPYALL = copy all file attributes
# /R:3 = retry 3 times on failure
# /W:5 = wait 5 seconds between retries
# /LOG = write log file for verification
# /TEE = show progress in console AND log
💡
MTP (phone USB) transfer speed is typically 15–40 MB/s. At 65GB, expect 30–70 minutes. Keep the phone screen on (set Display Timeout to "Never" in phone settings during transfer).
⚠️
Do NOT lock the phone or let it sleep during transfer — MTP connection drops on sleep. On Vivo: Settings → Display → Screen Timeout → set to "30 minutes" or max.

CRITICAL: Enroll MOK on Reboot (Blue Screen)

⚠
Windows time is wrong after switching from Ubuntu (Temporal Sync Bug)
COMMON ▶

Linux stores the hardware clock (RTC) in UTC format by default, whereas Windows 11 treats the hardware clock as local time. When dual-booting, this discrepancy results in a 6-hour offset (for Dhaka timezone, UTC+6) on every reboot.

🔴
Windows 11 24H2 Registry Bug: The standard registry modification (RealTimeIsUniversal) that forces Windows to use UTC fails to persist across hybrid shutdown and Fast Startup states in Windows 11 24H2. The registry key gets bypassed, resulting in recurring time drift.

The only permanent and robust solution is to configure Linux to use the local RTC standard instead, matching Windows' time system.

Terminal (Ubuntu) — Fix on Linux side
# Force Linux to store time in local RTC (aligning with Windows)
sudo timedatectl set-local-rtc 1 --adjust-system-clock

# Verify RTC status
timedatectl
ℹ️
Look for RTC in local TZ: yes in the output. A warning about local time will appear; this can be safely disregarded for dual-boot setups.
⚠

COMMON ▶

GRUB shows, Ubuntu starts loading GNOME 50, then screen stays black.

Ubuntu 26.04 is Wayland-only — nomodeset does NOT work (that kernel param only enables X11 fallback modesetting, which doesn't exist in 26.04). The fix is MOK + driver reinstall:

  3. Press F10 to boot — should get a working (software-rendered) Wayland session

⚠
MediaTek MT7921/MT7922 Wi-Fi PCIe Error -5 & Controller Deadlocks
CRITICAL ▶

The MediaTek MT7921 and MT7922 network cards on the Legion 5 are prone to severe PCIe link crashes. Under Linux, this manifests in the system kernel log (dmesg) as:

dmesg output
mt7921e 0000:04:00.0: ASIC revision: 79220001
mt7921e 0000:04:00.0: hardware init failed
mt7921e: probe of 0000:04:00.0 failed with error -5

This PCIe error is caused by a sleep protection register (0x18011100) deadlock where the card is trapped in a low-power S4 state. This is triggered when booting from Windows into Linux while Windows Fast Startup holds an ACPI lock on the PCIe lane.

Resolution Steps:

  1. Disable Windows Fast Startup (Mandatory): Boot into Windows → open powercfg.cpl (Power Options) → click "Choose what the power buttons do" → click "Change settings that are currently unavailable" → uncheck "Turn on fast startup" → click Save Changes.
  2. Embedded Controller (EC) Hard Reset: If the Wi-Fi card is missing entirely from both Windows Device Manager (yellow triangle or Code 45) and Linux lspci, the PCIe lane registers are hard-locked. Shut down the laptop, disconnect the power cord, and hold the power button down for exactly 60 seconds. This drains the EC capacitance and forces a fresh hardware enumeration.
  3. Stable Firmware Rollback: If running Linux kernel 6.14/6.16, the default linux-firmware package contains regressions causing command timeouts. Downgrading to the stable 20250311-1 firmware release resolves this.
Terminal
# Clone the official linux-firmware tree
git clone https://git.kernel.org/pub/scm/linux/kernel/git/firmware/linux-firmware.git ~/linux-firmware
cd ~/linux-firmware

# Copy stable MT7922 firmware binaries
sudo cp mediatek/WIFI_MT7922_patch_e2_hdr.bin /lib/firmware/mediatek/
sudo cp mediatek/WIFI_RAM_CODE_MT7922_1_1_hdr.bin /lib/firmware/mediatek/

# Rebuild initramfs boot image to pack the drivers
sudo update-initramfs -u
  1. Windows Driver Update: Under Windows 11, upgrade the MediaTek MT7921/MT7922 OEM driver to version 3.0.1.1198 or later to ensure proper power state transitions during warm reboots.
